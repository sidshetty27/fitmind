# FitMind API (backend)

FastAPI service. Python 3.11+.

## Local development

```bash
cd backend
python -m venv .venv
.venv\Scripts\activate          # Windows PowerShell: .venv\Scripts\Activate.ps1
pip install -r requirements.txt
cp .env.example .env            # then edit values — DATABASE_URL is required
alembic upgrade head            # create the schema + seed the exercise catalog
uvicorn app.main:app --reload --port 8000
```

- API: http://localhost:8000
- Interactive docs (Swagger): http://localhost:8000/docs
- Liveness: http://localhost:8000/health
- Readiness (database): http://localhost:8000/health/db

## Database (Phase 3)

Two connection URLs, because Supabase exposes two and they are not interchangeable:

| var | Supabase connection | why |
| --- | --- | --- |
| `DATABASE_URL` | Transaction pooler, **6543** | short per-request connections are what it is built for |
| `MIGRATION_DATABASE_URL` | Direct connection, **5432** | DDL needs session state pgbouncer cannot hold in transaction mode |

Against a plain local Postgres, set only `DATABASE_URL` — the migration URL falls back to it.

### Migrations

```bash
alembic upgrade head                              # apply
alembic downgrade -1                              # roll back one revision
alembic current                                   # what is applied
alembic revision --autogenerate -m "add x"        # after changing models
alembic upgrade head --sql                        # print SQL, connect to nothing
```

Always read an autogenerated revision before applying it. Autogenerate is a good diff engine, not a good judge — it does not reliably detect renames (it sees a drop plus an add, which loses data) or server-default changes it cannot introspect.

### Layout

```
app/
  core/config.py     settings; normalises any Postgres DSN onto psycopg3
  core/auth.py       Clerk JWT verification + get_current_user dependency
  core/clerk.py      Clerk Backend API client (email fallback for provisioning)
  core/svix.py       webhook signature verification (stdlib, no svix package)
  db/base.py         DeclarativeBase, constraint naming convention, mixins
  db/session.py      async engine, session factory, get_db dependency
  models/            User, Workout, Exercise, WorkoutExercise, ProgressEntry
  schemas/           Pydantic request/response contracts (separate from ORM)
  crud/              user, exercise, workout, progress — user-scoped data access
  api/routes/        health, me, exercises, workouts, progress, webhooks
alembic/versions/    0001 schema, 0002 exercise catalog seed
```

## Authentication (Phase 4)

The backend verifies the Clerk **session JWT** the frontend sends
(`Authorization: Bearer <token>`) against Clerk's published JWKS — no per-request
call to Clerk. Set `CLERK_ISSUER` (your Frontend API URL); JWKS is derived as
`{issuer}/.well-known/jwks.json`. See `.env.example` for every variable.

**Recommended: add an `email` claim to your Clerk JWT template.** Users are
just-in-time provisioned on first request, and `users.email` is NOT NULL. Give the
token the email so no extra call is needed:

1. Clerk Dashboard → **Sessions → Customize session token**.
2. Add claims: `{ "email": "{{user.primary_email_address}}", "name": "{{user.full_name}}" }`.

If you skip this, set `CLERK_SECRET_KEY` and the API fetches the email from Clerk's
Backend API as a fallback.

### User-sync webhook

Optional but recommended. Clerk Dashboard → **Webhooks** → add an endpoint at
`https://<your-api>/api/webhooks/clerk`, subscribe to `user.created`,
`user.updated`, `user.deleted`, and copy the **Signing Secret** into
`CLERK_WEBHOOK_SECRET`. Deliveries are Svix-signed and rejected if the signature
fails. JIT provisioning still covers correctness if the webhook is not configured.

### Endpoints

| method | path | purpose |
| --- | --- | --- |
| GET/PATCH | `/api/me` | the signed-in user's profile |
| GET | `/api/exercises`, `/api/exercises/{id}` | read the shared catalog (filters: `muscle_group`, `equipment`, `search`) |
| POST/GET | `/api/workouts` | create / list (filters: `date_from`, `date_to`, `limit`, `offset`) |
| GET/PATCH/DELETE | `/api/workouts/{id}` | one workout |
| PUT | `/api/workouts/{id}/exercises` | replace a workout's exercise list atomically |
| PUT/GET | `/api/progress` | upsert today / list a date range |
| GET/DELETE | `/api/progress/{recorded_on}` | one day's entry |
| POST | `/api/webhooks/clerk` | Clerk → user sync (Svix-signed) |

## Tests

```bash
pytest                                                    # no database needed
TEST_DATABASE_URL=postgresql://.../fitmind_test pytest    # + integration tests
```

The default run needs no database: the fixtures point at an unreachable one on
purpose, which is what makes the readiness endpoint's 503 path testable in CI, and
lets the auth layer's 401s and the webhook's signature checks be verified without
Postgres. The end-to-end CRUD tests in `test_api_integration.py` opt in via
`TEST_DATABASE_URL` (point it at a throwaway database — the schema is created and
dropped around each test) and skip otherwise.
